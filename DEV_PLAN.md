# DEV_PLAN — загальний план розвитку adaptive_aisa

Тримає всі лінії робіт і дві цілі в одному місці. Авторитетний журнал — `CLAUDE.md`;
концепт — `OVERVIEW.md`; цей файл — карта напрямків.

## Концепт (стисло)

Реальний оркестратор радить реальні стратегії → синтетичний AI-трейдер (7 персон) реагує →
на реакціях моделюється trust → trust переходить в утриманий оборот (revshare) → політики
дій оркестратора максимізують успіх. Замкнутий цикл advisor↔persona↔trust на полігоні
бэктест-кривих стратегій (groundtruth). Деталі — OVERVIEW.md.

## Три лінії робіт

### Лінія 1 — Adaptive Answer / дії оркестратора
Оркестратор відповідає по-різному залежно від типу питання і клієнта. У новому концепті це
не просто «окремий рекомендатель», а САМЕ ТІ дії оркестратора, які потім оптимізують політики
(лінія 3). Окупається сама, не залежить від партнера.
- **Intent-роутер.** Промпти під тип питання: "поясни" → без карток, "що включити" →
  ранжування, "чи входити зараз" → ризик-вердикт. Метрика: на ~20 заморожених питаннях
  частка відповідей у правильній формі ≥80%.
- Статус: спроєктовано, не реалізовано (TASK 03). Найдешевший ранній ефект.

### Лінія 2 — Дані: цикл advisor↔persona↔trust — ЦІЛЬ 1
Харнес моделює замкнутий цикл на полігоні й генерує розмічені рядки-точки з trust-динамікою.
- Статус: **частково. Шар 1 (персона+вікно) зібрано (TASK 02), АЛЕ концепт уточнено —**
  потрібна доробка: (а) джерело = бэктест-криві стратегій (qpnls через benchmark_sync), не
  голі OHLCV-вікна; (б) у цикл входить реальний оркестратор (радить стратегію за метриками);
  (в) trust як змінна, що оновлюється за розривом обіцяне↔пережите; (г) повний простір дій
  (тримати/докупити/перемкнутися/вийти/чат/мовчати), не лише вихід на просадці; (д) groundtruth
  з продовження кривої після T.
- Те, що ЗБЕРІГАЄТЬСЯ з TASK 02: точка-атом, відтворюваність за seed, gates, 7 персон,
  контракт спостережуваності, прихований вектор як службове поле.
- Те, що МІНЯЄТЬСЯ: джерело даних (OHLCV→qpnls), додається оркестратор у цикл, додається trust,
  розширюється простір дій, додається groundtruth-звірка.
- Наступне: TASK 05 (доробка харнесу під цикл), потім TASK 04 (gate на змістовність циклу).

### Лінія 3 — Політики — ЦІЛЬ 2
Поверх змодельованого циклу: які дії оркестратора максимізують Σ(trust → утриманий оборот → revshare).
Порядок строгий: цикл/дані → trust-динаміка → політики.
- Принцип: на кожен критичний trust-патерн — кандидат-дія оркестратора, A/B у симуляції
  (ті самі персони з/без дії), замір утриманого обороту.
- Відкриті питання (закрити перед лінією 3):
  - A/B без живих людей: ті самі персони + полігон, з дією оркестратора і без — порівняти утримання;
  - зв'язок дії з revshare: утриманий оборот = Σ позицій, що персона НЕ кинула × комісія;
  - перенос синтетики на реальних — звірка через user_id.
  Розрив ВУЖЧИЙ, ніж у 0.1: полігон дає groundtruth і A/B стає визначеним (та сама крива,
  різна дія оркестратора). Але остаточно не пророблено — не вважати закритим.

## Дві цілі
ЦІЛЬ 1 = дані/цикл (лінія 2). ЦІЛЬ 2 = політики (лінія 3). Друга на першій.
Лінія 1 = дії оркестратора, які лінія 3 оптимізує; окупається самостійно паралельно.

## Ключове допущення (не перевідкривати без причини)
Оркестратор радить за МЕТРИКАМИ стратегії, поза часом. Допущення: метрики стабільні на t-3Y…t.
Витоку майбутнього нема. Реал-тайм — тільки на боці персони. Розрив обіцяний профіль ↔ пережита
локальна ділянка кривої = паливо trust-динаміки. Деталі — OVERVIEW §Ключове допущення.

## Передумови
- `user_id` від WhiteBIT — умова валідації патернів на людях. Не перевідкривати.
- Зв'язок рекомендація→бэктест: pattern brute_{brute_id}_{settings_id}_{ASSET}USDT →
  dev.backtest_runs (is_benchmark=TRUE) → qpnls на remote scalping-pnl. Підтверджено в коді ATSA.
- Консенсус-таксономія (47→8-12) — передумова фінальних меж персон. Відкладена (20%-хвіст).

## Точка сходу — демо для Dmitriy (WhiteBIT)
Персони отримують рекомендації оркестратора, проживають криві стратегій, trust зсувається,
частина йде в silent churn. Демо показує: рекомендація → trust → утриманий/втрачений оборот → комісія.
Рамка: proof of plausibility (для розмови про дані/доступ), не proof of effect до живих людей.

3. **TASK 04 — gate:** чи відтворює цикл змістовну trust-динаміку; чи різні причини
   розрізнювані за полями. Прогнати раз, зафіксувати.
4. Консенсус-таксономія і точний 12-код режиму — відкладені (20%-хвіст), лише при калібруванні.
5. Лінія 3: політики поверх trust-динаміки — після кроків 2-3.
