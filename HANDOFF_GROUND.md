# HANDOFF — підготувати ґрунт для швидкої, плідної, аудованої роботи

Прочитати: `concept/00_PITCH.md` (ЯКІР: one-liner, пітч, bandit-not-POMDP) → цей файл → `CLAUDE.md` (журнал, append-only) → `DEV_PLAN.md`.

**ФОКУС (00_PITCH):** синтетичні трейдери для дослідження попиту і тренування політик маркетингу через адаптивні рекомендації AI-Адвайзера. trust = ФІЧІ з наблюдаемої історії, НЕ сховане стан. Політики = contextual bandit, НЕ POMDP. Інквізитор опускає на землю цим щосесії.
Це завдання НЕ на код циклу. Це завдання — створити інфраструктуру контролю, якої бракує.
Власник не має часу на прозу. Він хоче бачити простими словами: рухаємось до істини чи ні.

## Контекст (стан на 2026-06-23)
- Концепт стабільний: оркестратор радить → персона проживає криву → розрив обіцяне↔пережите
  рухає trust (вектор satisfaction/dependency/momentum) → trust → утриманий оборот → revshare.
- STOP_TEST закрито з уточненням ролі: це був GO/NO-GO на ПЕРЕДУМОВИ, не тест фальсифікованості.
  Передумови виконані: розрив у даних існує (топ-3 інверсія=100%), groundtruth існує (криві 2-3р).
  Вердикт: GO. rho=0.323/поріг 0.3 — НЕ вердикт стопу, а побічний факт про оркестратор.
- Справжня фальсифікованість концепту перевіряється ВСЕРЕДИНІ харнесу (перший Gate TASK 04):
  чи узгоджені trust-реакції з фактом кривої. Цей Gate ОБОВ'ЯЗКОВО мусить мати змогу провалитись.

## ЩО ЗРОБИТИ (4 артефакти, Парето — мінімум що дає контроль)

### 1. PROGRESS.md — єдине джерело «де ми»
Таблиця: етап | критерій DONE (бінарний) | статус | дата | посилання на доказ.
Рядок на кожен TASK + кожен Gate. Один погляд = розуміння стану. СС оновлює 1 рядок за сесію.
НЕ проза. НЕ переказ. Тільки таблиця.

### 2. CHECKLIST.md — гейти як питання так/ні
Для кожного Gate (TASK 04 A/B, контракт спостережуваності, seed-відтворюваність,
A/B-горизонт через зміну режиму) — список перевірок у форматі бінарних питань.
Кожне питання мусить мати ЗМОГУ дати «ні» (інакше гейт фіктивний — головний урок інквізиції).

### 3. GROUNDTRUTH.md — зафіксувати істину як артефакт
Зараз groundtruth лише описаний, не зафіксований. Визначити ОДНОЗНАЧНО:
- ЩО є істина: продовження кривої після T (напр. фактичний ROI через k чекпоінтів).
- ДЕ лежить: точний шлях/таблиця/поле (звірити з кодом, не з пам'яттю).
- ЯК звіряється: процедура «дія персони vs факт кривої» (вихід виправданий якщо крива далі впала).
- КРИТИЧНО: та сама крива, яку персона «проживає», = та, з якої береться факт. Інакше звірка хибна.

### 4. Формат звіту СС (SESSION_REPORT_TEMPLATE.md)
Кожна сесія СС ЗАВЕРШУЄТЬСЯ цим, не прозою:
- Перевірено: <що саме, відтворювано>
- Підтвердилось / Спростувалось: <бінарно, з числом>
- Рух до істини: ТАК / НІ / ЧОМУ (одне речення)
- Наступний фальсифікований крок: <що може провалитись далі>
- Оновлено PROGRESS.md рядок: <який>

## ОБОВ'ЯЗКОВА ПЕРЕВІРКА ПЕРЕД БУДЬ-ЧИМ (борг із сесії)
Ці два не закриті, блокують достовірність майбутніх гейтів:

- Чи та сама крива, що йде в харнес. Якщо ні — groundtruth-звірка буде хибна.

### P2. Зафіксувати в CLAUDE.md уточнення статусу STOP_TEST
Поточний запис подає rho=0.323/BRANCH 1 як вердикт. Це не вердикт стопу.
Додати рядок: STOP_TEST = GO/NO-GO на передумови (GO); rho/поріг/fitness — побічні факти
для рівня 3 (політики), не критерій старту; справжня фальсифікованість — перший Gate харнесу.
НЕ переписувати історію (append-only) — додати уточнюючий запис.
