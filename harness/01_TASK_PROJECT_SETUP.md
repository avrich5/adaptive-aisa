# 01_TASK_PROJECT_SETUP — виніс adaptive_aisa у самостійний репозиторій

**Тип:** план для майбутньої код-сесії (Claude Code). НЕ виконання в цій сесії.
**Крок EXECUTION_ORDER:** передумова до реалізації (крок 3), не блокує написання спек (кроки 1-2).
**Ціль:** винести `adaptive_aisa/` з-під `wbprd_macbook` у власний git-репозиторій
`https://github.com/avrich5/adaptive-aisa.git`. Зараз папка живе без git усередині `wbprd_macbook`.

---

## Рішення Андрія, що фіксують рамку (не перевідкривати)

- Новий репозиторій: `https://github.com/avrich5/adaptive-aisa.git`.
- Переміщення файлів — звичайним `mv`, не `git mv` (git усередині `adaptive_aisa/` ще нема).
- README стає точкою входу замість `00_INDEX.md`.
- `CLAUDE_CODE_TASK_CONSENSUS.md` лишається в корені, не архівувати.
- `_archive/` не чіпати без причини.

---

## Що спланувати (вихід цього таску)

### 1. Структура цільового репо

Запропонувати розкладку. Орієнтир (уточнити при виконанні, звіривши з реальним станом папки):

```
adaptive-aisa/
  README.md                      точка входу, замінює 00_INDEX.md
  concept/                       00_CONCEPT.md, 00_OPEN_QUESTIONS.md
  inventory/                     A_–H_, sql_log, D_user_observations, E_gap_analysis
  personas/                      experts, _merged, _report, consensus (коли зʼявиться)
  harness/                       вміст synthetic_trader_harness/ (спеки + реалізація)
  tasks/                         CLAUDE_CODE_TASK_*.md
  _archive/                      перенести як є, не реорганізовувати
  scripts/                       consolidate_experts.py та інші
```

Рішення прийняти при виконанні, не зараз: чи зберігати префікси-нумерацію файлів
(`00_`, `A_`) після переходу на теки, чи перейти на чисті імена з порядком через README.

### 2. Що переносити, а що лишати посиланням

Інвентаризаційне ядро (A_–H_, sql_log, personas, experts, scripts) переноситься повністю.
Звірити, чи якісь файли в `adaptive_aisa/` посилаються на шляхи поза `adaptive_aisa/`
(на `pr-platform-architecture`, `follow-docs`, БД-конфіги). Такі посилання НЕ розривати:
лишити відносним посиланням або задокументувати залежність у README, а не копіювати чужі файли.

### 3. `.gitignore`

Виключити:
- `*.parquet` (`all_regimes.parquet` і похідні — великі, git-ignored і в оригіналі);
- креди й секрети (`.env`, ключі, конекшн-стрінги до `77.42.77.82`, VPN-конфіги);
- великі JSON-дампи стратегій / бенчмарків, якщо лежать у папці;
- локальні артефакти прогонів харнесу (траєкторії, кеші) — додати теку `runs/` чи `output/` в ignore;
- `__pycache__/`, `.DS_Store`, віртуальні оточення.

Перед першим комітом — `git status` і ручна перевірка, що жоден креденшіал/великий бінарник
не потрапляє в індекс.

### 4. README як точка входу

Перенести з `00_INDEX.md` навігацію, додати:
- однофразовий опис проєкту (з `00_CONCEPT.md` §1);
- порядок читання (HANDOFF → EXECUTION_ORDER → CONCEPT → D/E);
- рамку «scenario-tool і тренажер політики, НЕ предиктор» (HANDOFF п.5);
- статус (фаза 0 роадмапу).

### 5. git init + перший коміт

- `git init` у винесеній теці;
- `remote add origin https://github.com/avrich5/adaptive-aisa.git`;
- перевірка `.gitignore` (п.3) ДО `git add`;
- перший коміт — імпорт існуючого стану як є, без рефактору вмісту;
- НЕ форсити, НЕ переписувати історію (історії ще нема).

## Послідовність виконання (для код-сесії)

1. Зняти реальний інвентар `adaptive_aisa/` (`ls -R`, розмір файлів, наявність бінарників).
2. Узгодити з Андрієм фінальну структуру тек і долю нумерації імен.
3. Створити цільову теку, `mv` файли за погодженою розкладкою.
4. Написати `.gitignore` і `README.md`.
5. `git init` + перевірка статусу + перший коміт + push.
6. Звірити, що жоден внутрішній лінк не зламався після переміщення (grep по старих шляхах).

## Підводні камені

- **Биті посилання після `mv`.** У папці вже був випадок битого указівника
  (`00_HUMAN_NARRATIVE.md` → `00_CONCEPT.md`). Після будь-якого переміщення — grep по іменах
  файлів у всіх `.md`, поправити відносні шляхи.
- **Не тягнути чужі файли в репо.** Якщо `adaptive_aisa/` посилається на артефакти
  `pr-platform-architecture` чи прод-сервісів — це залежність, не вміст. Документувати, не копіювати.
- **Паркети й креди.** Головний ризик першого коміту — затягнути великий `.parquet` або `.env`.
  Перевірка `.gitignore` строго до `git add`.
- **Це план.** Реальні `mv`/`git init` — у код-сесії, не тут.
