## Gaps identified (POMDP perspective)

### Gap 1 — `regime_performance` top-level not consumed
The v7.3 field `data.regime_performance` contains structured per-regime metrics (R1–R12 keyed dict). Only 4161/16709 strategies have it populated. The orchestrator does **not read it**. Regime-awareness is derived from `market_context.benchmark_periods` instead, which is coarser (regime labels from historical intervals, not labeled base-states R1-R12 tags).

For POMDP strategy profiles (per-regime expected return + risk), `regime_performance` is the correct source — but it needs:
1. Broader coverage (currently 24.9%)
2. Orchestrator code to read it

### Gap 2 — `benchmark` column coverage: 62.3%
6291 strategies have no `benchmark` column. For those, `avg_roi` falls back to `aggregate_stats.avg_roi` and WB risk filters use `mdd_avg` from unified JSON Performance string (regex fallback). Per-regime data in `benchmark` is [UNKNOWN — requires clarification of benchmark column schema].

### Gap 3 — `wb_context` coverage: 51.2%
8161 strategies have no `wb_context`. `regime_risk_flags.negative_in[]` — the only current regime-conditional filtering in the orchestrator — is absent for those strategies.

### Gap 4 — `benchmark_periods` regime labels
`benchmark_periods` maps intervals to regime labels using a simple year-based rule in the generator (`2022 → bear`, `2023/2024 → bull`, else `ranging`). This is NOT base-states R1-R12. The mapping to R1-R12 happens in `compute_strategy_regime_fit` via `short_code_to_full` in `regime_mappings.py`. Correctness of this mapping depends on whether benchmark interval dates align with actual regime boundaries.

### Gap 5 — No per-regime Sharpe or drawdown in orchestrator retrieval
The orchestrator retrieval and sorting use aggregate `avg_roi`/`avg_fitness` only. Per-regime Sharpe or drawdown is not part of any filter or sort key currently.
